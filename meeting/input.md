# 社区会议主动管理机制设计

## 任务目标

作为开源社区运营人员，设计并实现一套**社区会议主动管理机制**，通过以下手段提升社区会议的正常运转率和社区健康度：

1. **事前预防**：在会议到期前主动提醒、跟踪，避免会议被遗忘或延期
2. **过程监控**：实时掌握各 SIG/TC/TSC 会议的召开情况和质量
3. **事后治理**：对未达标的 SIG 或个人，触发整改或角色调整流程
4. **数据驱动**：通过仪表盘和报表量化会议健康指标，辅助运营决策，此项无需进行代码开发，由已有平台提供数据

## 关键问题

1. **完全依赖 Maintainer 主动性**：无人提醒时会议容易被遗忘
2. **缺乏统一监控视图**：运营人员无法一眼看到哪些 SIG 本月还没开会
3. **没有预警机制**：只能事后发现会议未召开，无法事前介入
4. **纪要/议程质量参差不齐**：没有标准化检查
5. **角色不活跃识别延迟**：Maintainer 3 个月不参会才触发不活跃判定，缺乏中间状态预警

## 输入数据

### 数据源 1：组织人员信息与运作规则

来源：社区community仓库下，各组织文件中 `org-info.yaml` 和 `sigs/*/sig-info.yaml`

- 项目列表：AscendNPU-IR, common, FrameworkPTAdapter, MindCluster, MindIE, MindSeriesSDK, MindSpeed, MindStudio, TSC
- 每个项目的 TC 成员列表（`tc_members`）
- 每个 SIG 的 Maintainer 列表（`org-info.yaml` → `sigs[].maintainers`）
- 每个 SIG 的 Committer/Reviewer 列表（`sig-info.yaml` → `committers` / `reviewers`）
- SIG 的 meeting_url 字段
- 组织运作规则文件，用于提取运作标准和频率，（如未查询到可向我提问）

### 数据源 2：会议记录

来源：Ascend Meeting 中心提供查询接口，仅需实现调用对应接口获取数据能力，接口支持如下查询：

- 各 SIG 的会议预约记录（时间、主题、参会人）
- 各 TC/TSC 会议的预约记录
- 实际参会人数和出席名单

### 数据源 3：会议纪要

来源：Etherpad，社区基础设施提供查询接口，仅需实现调用对应接口获取数据能力，接口支持如下查询：

- 各 SIG 对应的 Etherpad pad 内容

## 输出要求

### 输出 1：社区会议健康度报表

一个可视化页面，包含：

| 模块 | 内容 | 刷新频率 |
|------|------|----------|
| **全局概览** | 本月 SIG 会议召开率、TC/TSC 会议召开率、议程发布率、纪要归档率 | 每日 |
| **SIG 明细** | 每个 SIG 上次开会时间、下次预约时间、距离月底剩余天数、Maintainer 信息 | 每日 |
| **风险预警** | 本月未开会且剩余时间 ≤7 天的 SIG 列表（红灯）、≤14 天（黄灯） | 每日 |
| **TC/TSC 追踪** | 各 TC 本年度已开会次数 vs 要求次数（≥4）、成员出席率 | 每周 |
| **个人活跃度** | Maintainer 最近 3 个月参会记录，标记疑似不活跃人员 | 每月 |

### 输出 2：自动提醒消息

| 事件 | 触发条件 | 提醒对象 | 渠道 | 内容模板 |
|------|----------|----------|------|----------|
| **月初提醒** | 每月 1 日 | 所有 SIG Maintainer | 邮件 + 群消息 | "请安排本月 {SIG名} 例会，截止日期 {月底}" |
| **中期预警** | 每月 15 日，仍未预约本月会议的 SIG | 对应 Maintainer + TC Chair | 邮件 + 群消息 | "⚠️ {SIG名} 本月尚未预约例会，请尽快安排" |
| **紧急预警** | 每月 22 日，仍未预约的 SIG | Maintainer + TC Chair + 运营 | 邮件 + 群消息 | "🔴 {SIG名} 会议风险：距月底仅剩 {N} 天" |
| **议程缺失提醒** | 会议预定日 T-2 天，未发布议程 | Maintainer | 邮件 + 群消息 | "会议 {日期} 的议程尚未发布到 Etherpad" |
| **纪要催办** | 会议结束 T+2 天，纪要未归档 | Maintainer | 邮件 | "请归档 {日期} 会议纪要" |
| **TC 季度检查** | 每季度末 | TC Chair | 邮件 | "本季度已召开 {N} 次会议，年度目标 ≥4 次" |
| **不活跃预警** | 连续 2 个月未参加 SIG 会议 | Maintainer 本人 + TC Chair | 邮件 | "根据社区规则，连续 3 个月不参会将触发不活跃判定" |

发送提醒后，需在下次报告中记录已发送提醒次数，以及提醒收件人。

### 输出 3：月度/季度运营报告

Markdown 格式报告，包含：

```
## 社区会议运营报告 — {YYYY}年{MM}月

### 总览
- SIG 例会召开率：{X}%
- 议程提前发布率：{X}%
- 纪要按时归档率：{X}%
- TC 例会召开率：{X}%

### 各 SIG 明细
| SIG | 所属项目 | 是否召开 | 会议日期 | 议程 | 纪要 | Maintainer |
|-----|----------|----------|----------|------|------|------------|
| ... | ...      | ✅/❌    | ...      | ✅/❌ | ✅/❌ | ...       |

### 风险项
- {列出未达标 SIG 及原因}

### 改进建议
- {基于数据的运营建议}
```


## 处理流程

### 阶段一：数据基础建设

1. **标准化 meeting_url 字段**：推动各 SIG 在 `sig-info.yaml` 中填写真实的 Etherpad 纪要链接，制定 PR 模板和检查规则
2. **建立会议注册表**：创建一个中心化的 `meeting-registry.yaml` 或数据库，记录每个 SIG 的预期会议频率、会议日（如"每月第二个周三"）、Etherpad pad 名称、通知群组
3. **打通 Meeting Center 数据**：确认 Ascend Meeting Center 是否提供 API；如无，设计定期数据抓取方案

### 阶段二：监控与预警系统

1. **开发会议状态轮询服务**：每日检查各 SIG 本月是否已预约/已召开会议，更新状态
2. **实现分级提醒引擎**：按上述提醒规则表，在触发条件满足时通过邮件/群消息通知相关人
3. **建设仪表盘**：实现全局概览、SIG 明细、风险预警三个核心视图

### 阶段三：治理流程闭环

1. **定义升级流程**：
   - 第 1 次未达标 → 邮件提醒 + 记录
   - 连续 2 次未达标 → 运营介入，与 Maintainer 沟通原因
   - 连续 3 次未达标 → 触发 TC 审议，考虑 Maintainer 角色调整（对应社区规则"3 个月不活跃"）
2. **制定议程/纪要质量标准**：定义最低内容要求（如：议程需包含至少 1 个讨论议题；纪要需包含出席人、讨论摘要、决议事项）
3. **自动月报生成**：每月 1 日自动生成上月报告，发送给 TSC 和各 TC Chair

### 阶段四：持续优化

1. **收集反馈**：每季度征集 Maintainer 对提醒机制的反馈，调整频率和措辞
2. **扩展指标**：增加会议参与人数趋势、议题完成率等进阶指标

## 示例

### 示例 1：SIG 月度未开会预警流程

**场景**：MindSpeed-LLM SIG 在 2026 年 3 月未预约任何会议

**流程**：
1. 3 月 1 日 → 系统发送月初提醒给 MindSpeed-LLM 的所有 Maintainer
2. 3 月 15 日 → 系统检测到无预约记录，发送中期预警给 Maintainer + MindSpeed TC Chair
3. 3 月 22 日 → 仍无预约，发送紧急预警给 Maintainer + TC Chair + 运营团队
4. 运营介入：电话或私信联系 Maintainer，了解原因，协助安排会议
5. 3 月 31 日 → 若仍未召开，记录为"未达标"，纳入月报
6. 若连续 3 个月未达标 → 运营提交议题给 TC，建议审议 Maintainer 活跃度
