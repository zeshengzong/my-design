# 社区会议主动管理机制设计

## 任务目标

作为开源社区运营人员，设计并实现一套**社区会议主动管理机制**，通过以下手段提升社区会议的正常运转率和社区健康度：

1. **事前预防**：在会议到期前主动提醒、跟踪，避免会议被遗忘或延期
2. **过程监控**：实时掌握各组织（SIG、TC、TSC、理事会、专家委员会、用户委员会、技术治理委员会等）会议的召开情况和质量
3. **事后治理**：对未达标的组织或个人，触发整改或角色调整流程
4. **数据驱动**：通过仪表盘和报表量化会议健康指标，辅助运营决策，此项无需进行代码开发，由已有平台提供数据

## 关键问题

1. **完全依赖 Maintainer 主动性**：无人提醒时会议容易被遗忘
2. **缺乏统一监控视图**：运营人员无法一眼看到哪些组织本月还没开会
3. **没有预警机制**：只能事后发现会议未召开，无法事前介入
4. **纪要/议程质量参差不齐**：没有标准化检查
5. **角色不活跃识别延迟**：Maintainer 3 个月不参会才触发不活跃判定，缺乏中间状态预警
6. **组织覆盖不全**：当前仅关注 SIG/TC，未覆盖理事会、专家委员会、用户委员会、技术治理委员会等其他社区组织
7. **缺少组织级运营对接人**：出现会议风险时，没有明确的社区运营人员负责推进各组织的会议召开

## 输入数据

### 数据源 1：组织人员信息与运作规则（MCP Server）

社区包含多种组织类型：**SIG、TC、TSC、理事会、专家委员会、用户委员会、技术治理委员会**等，且未来可能新增其他组织类型。为统一获取完整的组织结构和人员数据，采用 **Model Context Protocol（MCP）** 方案，由社区基础设施侧提供 MCP Server。

#### 方案说明

社区基础设施团队实现一个 **Community Organization MCP Server**，遵循 [Model Context Protocol](https://modelcontextprotocol.io/) 规范，将社区组织数据封装为标准化的 Tool 供本系统（及 AI Agent）调用。

**MCP Server 需暴露的核心 Tools：**

| Tool 名称 | 描述 | 参数 | 返回 |
|-----------|------|------|------|
| `list_org_types` | 列出社区所有组织类型 | 无 | 组织类型列表，如 `["board", "expert_committee", "user_committee", "tsc", "tc", "sig", ...]` |
| `list_organizations` | 列出指定类型下的所有组织 | `org_type: string` | 组织列表，含名称、ID、所属项目等 |
| `get_org_detail` | 获取单个组织的详细信息 | `org_id: string` | 成员列表（含角色）、会议频率要求、meeting_url、Etherpad 链接等 |
| `get_org_members` | 获取组织成员及角色 | `org_id: string` | 成员列表，含 `role`（chair / maintainer / member 等） |
| `get_meeting_rules` | 获取组织的会议运作规则 | `org_id: string` | 会议频率、议程要求、纪要要求等 |

**MCP 可行性评估：**

- ✅ **可行**：MCP 协议本身是轻量级 JSON-RPC over HTTP/SSE，社区基础设施只需实现一个服务端，将已有的 `org-info.yaml`、`sig-info.yaml` 以及其他组织数据封装为 Tool 即可
- ✅ **解耦**：本系统不依赖特定数据存储格式（YAML/数据库），只面向 MCP Tool 接口编程
- ✅ **可扩展**：新增组织类型（如未来增加"工作组"等）只需 MCP Server 侧更新数据，本系统零改动
- ✅ **AI 友好**：天然支持 AI Agent 通过 MCP 查询组织数据并做智能分析
- ⚠️ **前置依赖**：需社区基础设施团队配合开发 MCP Server，建议先实现只读 Tools，后续可扩展写入能力

#### 降级方案

当 MCP Server 不可用时，系统降级为直接读取社区 community 仓库下的 YAML 文件：

- `org-info.yaml`：获取项目列表、TC 成员、SIG Maintainer 等
- `sigs/*/sig-info.yaml`：获取 SIG 的 Committer/Reviewer、meeting_url 等
- 组织运作规则文件：提取运作标准和频率

> 注：降级方案仅覆盖 TC/SIG 数据，理事会、专家委员会等组织的数据必须通过 MCP Server 获取。

### 数据源 2：会议记录

来源：Ascend Meeting 中心提供查询接口，仅需实现调用对应接口获取数据能力，接口支持如下查询：

- 各组织（SIG、TC、TSC、理事会、专家委员会、用户委员会、技术治理委员会等）的会议预约记录（时间、主题、参会人）
- 实际参会人数和出席名单

### 数据源 3：会议纪要

来源：Etherpad，社区基础设施提供查询接口，仅需实现调用对应接口获取数据能力，接口支持如下查询：

- 各组织对应的 Etherpad pad 内容

## 输出要求

### 输出 1：社区会议健康度报表

一个可视化页面，包含：

| 模块 | 内容 | 刷新频率 |
|------|------|----------|
| **全局概览** | 本月各类组织会议召开率（SIG、TC/TSC、理事会、各委员会）、议程发布率、纪要归档率 | 每日 |
| **组织明细** | 每个组织上次开会时间、下次预约时间、距离截止日剩余天数、负责人信息、对接社区运营 | 每日 |
| **风险预警** | 本月未开会且剩余时间 ≤7 天的组织列表（红灯）、≤14 天（黄灯） | 每日 |
| **TC/TSC 追踪** | 各 TC 本年度已开会次数 vs 要求次数（≥4）、成员出席率 | 每周 |
| **委员会追踪** | 理事会、专家委员会、用户委员会、技术治理委员会的会议召开情况 vs 要求频率 | 每月 |
| **个人活跃度** | Maintainer / 委员会成员最近 3 个月参会记录，标记疑似不活跃人员 | 每月 |

### 输出 2：自动提醒消息

> 说明：以下提醒规则以 SIG（月度例会）为典型示例。系统根据各组织在 `meeting-registry.yaml` 中配置的会议频率（月度/季度/半年度）自动计算触发时间点。每个组织均配置有对接的**社区运营人员**（`operators` 字段），风险预警时由社区运营负责推进会议召开。

| 事件 | 触发条件 | 提醒对象 | 渠道 | 内容模板 |
|------|----------|----------|------|----------|
| **周期初提醒** | 会议周期起始日（如月度组织每月 1 日，季度组织每季度首日） | 组织负责人（Maintainer / Chair / 委员等） | 邮件 + 群消息 | "请安排本周期 {组织名} 例会，截止日期 {周期末}" |
| **中期预警** | 周期过半仍未预约会议 | 组织负责人 + 社区运营 | 邮件 + 群消息 | "⚠️ {组织名} 本周期尚未预约例会，请尽快安排" |
| **紧急预警** | 距周期末 ≤7 天仍未预约 | 组织负责人 + 社区运营 | 邮件 + 群消息 | "🔴 {组织名} 会议风险：距截止日仅剩 {N} 天" |
| **议程缺失提醒** | 会议预定日 T-2 天，未发布议程 | 组织负责人 | 邮件 + 群消息 | "会议 {日期} 的议程尚未发布到 Etherpad" |
| **纪要催办** | 会议结束 T+2 天，纪要未归档 | 组织负责人 | 邮件 | "请归档 {日期} 会议纪要" |
| **TC/委员会周期检查** | 每季度末 | TC Chair / 委员会主任 + 社区运营 | 邮件 | "本季度已召开 {N} 次会议，年度目标 ≥{M} 次" |
| **不活跃预警** | 连续 2 个周期未参加组织会议 | 成员本人 + 社区运营 | 邮件 | "根据社区规则，连续 3 个周期不参会将触发不活跃判定" |

发送提醒后，需在下次报告中记录已发送提醒次数，以及提醒收件人。社区运营收到预警后应主动联系组织负责人，了解原因并协助推进会议召开。

## 处理流程

### 阶段一：数据基础建设

1. **接入 Community Organization MCP Server**：与社区基础设施团队协作，推动实现 MCP Server，提供组织结构、人员、会议规则的标准化查询能力；同步实现降级方案（直接读取 YAML 文件）
2. **标准化 meeting_url 字段**：推动各组织在对应配置文件中填写真实的 Etherpad 纪要链接，制定 PR 模板和检查规则
3. **建立会议注册表**：创建一个中心化的 `meeting-registry.yaml` 或数据库，记录每个组织的以下信息：
   - 组织 ID、类型（sig / tc / tsc / board / expert_committee / user_committee 等）、所属项目
   - 预期会议频率（monthly / quarterly / semi-annual）、会议日（如"每月第二个周三"）
   - Etherpad pad 名称、通知群组
   - **社区运营人员名单**（`operators`）：配置负责该组织会议推进的社区运营人员，含姓名、邮箱等联系方式

   示例结构：
   ```yaml
   organizations:
     - org_id: "sig-mindspeed-llm"
       org_type: "sig"
       parent_project: "MindSpeed"
       meeting_frequency: "monthly"
       meeting_schedule: "每月第二个周三 14:00"
       etherpad_pad: "sig-mindspeed-llm"
       notification_group: "mindspeed-sig-group"
       operators:
         - user_id: "op_user_1"
           name: "张三"
           email: "zhangsan@example.com"
         - user_id: "op_user_2"
           name: "李四"
           email: "lisi@example.com"

     - org_id: "expert-committee"
       org_type: "expert_committee"
       parent_project: null
       meeting_frequency: "quarterly"
       meeting_schedule: "每季度第一个月第二周"
       etherpad_pad: "expert-committee"
       notification_group: "expert-committee-group"
       operators:
         - user_id: "op_user_3"
           name: "王五"
           email: "wangwu@example.com"
   ```
4. **打通 Meeting Center 数据**：确认 Ascend Meeting Center 是否提供 API；如无，设计定期数据抓取方案

### 阶段二：监控与预警系统

1. **开发会议状态轮询服务**：每日检查各组织本周期内是否已预约/已召开会议，更新状态（通过 MCP Server 获取组织列表和会议规则，通过 Meeting Center 接口获取会议记录）
2. **实现分级提醒引擎**：按上述提醒规则表，在触发条件满足时通过邮件/群消息通知相关人；提醒对象从 `meeting-registry.yaml` 的 `operators` 字段读取社区运营人员，确保风险预警准确送达对接运营
3. **建设仪表盘**：实现全局概览、组织明细、风险预警、委员会追踪等核心视图

### 阶段三：治理流程闭环

1. **定义升级流程**：
   - 第 1 次未达标 → 邮件提醒 + 记录
   - 连续 2 次未达标 → 社区运营介入，联系组织负责人沟通原因
   - 连续 3 次未达标 → 社区运营上报给对应上级组织审议（如 SIG 上报 TC，TC 上报 TSC），考虑角色调整（对应社区规则"3 个月不活跃"）
2. **制定议程/纪要质量标准**：定义最低内容要求（如：议程需包含至少 1 个讨论议题；纪要需包含出席人、讨论摘要、决议事项）
3. **自动月报生成**：每月 1 日自动生成上月报告，发送给 TSC、各 TC Chair、各委员会主任及对接社区运营

### 阶段四：持续优化

1. **收集反馈**：每季度征集 Maintainer 对提醒机制的反馈，调整频率和措辞
2. **扩展指标**：增加会议参与人数趋势、议题完成率等进阶指标

## 示例

### 示例 1：SIG 月度未开会预警流程

**场景**：MindSpeed-LLM SIG 在 2026 年 3 月未预约任何会议

**流程**：
1. 3 月 1 日 → 系统发送周期初提醒给 MindSpeed-LLM 的所有 Maintainer
2. 3 月 15 日 → 系统检测到无预约记录，发送中期预警给 Maintainer + 社区运营（从 `meeting-registry.yaml` 的 `operators` 字段获取）
3. 3 月 22 日 → 仍无预约，发送紧急预警给 Maintainer + 社区运营
4. 社区运营介入：电话或私信联系 Maintainer，了解原因，协助安排会议
5. 3 月 31 日 → 若仍未召开，记录为"未达标"，纳入月报
6. 若连续 3 个月未达标 → 社区运营上报议题给 TC，建议审议 Maintainer 活跃度
