# 社区会议主动管理机制设计

## 任务目标

作为开源社区运营人员，设计并实现一套**社区会议主动管理机制**，通过以下手段提升社区会议的正常运转率和社区健康度：

1. **事前预防**：在会议到期前主动提醒、跟踪，避免会议被遗忘或延期
2. **过程监控**：实时掌握各组织（SIG、TC、TSC、理事会、专家委员会、用户委员会、技术治理委员会等）会议的召开情况和质量
3. **事后治理**：对未达标的组织或个人，触发整改或角色调整流程
4. **数据驱动**：通过仪表盘和报表量化会议健康指标，辅助运营决策，此项无需进行代码开发，由已有平台提供数据

## 关键问题

1. **完全依赖 Maintainer 主动性**：无人提醒时会议容易被遗忘
2. **缺乏统一监控视图**：运营人员无法一眼看到哪些组织本月还没开会
3. **没有预警机制**：只能事后发现会议未召开，无法事前介入
4. **纪要/议程质量参差不齐**：没有标准化检查
5. **角色不活跃识别延迟**：Maintainer 3 个月不参会才触发不活跃判定，缺乏中间状态预警
6. **组织覆盖不全**：当前仅关注 SIG/TC，未覆盖理事会、专家委员会、用户委员会、技术治理委员会等其他社区组织
7. **缺少组织级运营对接人**：出现会议风险时，没有明确的社区运营人员负责推进各组织的会议召开
8. **会议关键节点缺乏度量**：会议从预定到纪要归档有多个管理节点，但缺乏对每个节点是否按时完成的系统化跟踪

## 会议生命周期管理

每次组织会议的召开包含 **4 个关键管理节点**，系统需对每个节点的完成时间进行度量，未按时完成的节点触发对应提醒。会议关键节点包括：1.会议预订，2.发送征集议题邮件（社区邮件列表获取），3.召开会议，4.会议纪要。

### 生命周期节点定义

```
周期开始                                    会议日 T                周期截止
  │                                           │                      │
  ▼                                           ▼                      ▼
  ┌──────────-┐    ┌──────────────┐   ┌────────--─┐   ┌──--────────┐
  │ ① 预定会议 │──▶│ ② 征集议题邮件 │──▶│ ③ 召开会议 │──▶│ ④ 会议纪要 │
  └──────────-┘    └──────────────┘   └────────--─┘   └────--──────┘
   截止：T-7       截止：T-5          截止：T 当日     截止：T+3
   状态：PLANNED   状态：TOPICS_SENT  状态：HELD      状态：MINUTED
```

### 各节点详细规则

| 节点 | 动作 | 期望完成时间 | 检测方式 | 未达成时的状态 |
|------|------|-------------|----------|--------------|
| **① 预定会议** | 在 Meeting Center 中创建会议预约 | 会议日 T-7 天前 | 查询 Meeting MCP Server 是否存在预约记录 | `UNPLANNED` |
| **② 征集议题邮件** | 向组织邮件列表发送议题征集邮件 | 会议日 T-5 天前 | 查询 Mailing List MCP Server 是否有匹配邮件 | `TOPICS_MISSING` |
| **③ 召开会议** | 按预约时间正常召开会议 | 会议日 T 当日 | 查询 Meeting MCP Server 的出席记录 | `NOT_HELD` |
| **④ 会议纪要** | 将会议纪要归档到 Etherpad | 会议日 T+3 天内 | 查询 MCP Server 的 Etherpad 内容 | `MINUTES_MISSING` |

### 会议状态机

每次会议实例的完整状态流转：

```
UNPLANNED → PLANNED → TOPICS_SENT → HELD → MINUTED
                                      ↓
                                   OVERDUE（周期截止仍未完成当前节点）
```

- 系统每日轮询检查各组织当前周期的会议实例处于哪个节点
- 任一节点超过期望完成时间仍未达成，立即触发对应提醒
- 周期截止时，会议实例最终状态即为该周期的绩效结果

### 节点完成率度量

系统为每个组织计算以下指标（按月/季度统计）：

| 度量指标 | 计算方式 | 健康阈值 |
|----------|---------|---------|
| **预约及时率** | 在 T-7 前完成预约的会议占比 | ≥ 90% |
| **议题征集率** | 在 T-5 前发出征集邮件的会议占比 | ≥ 80% |
| **会议召开率** | 按计划召开的会议占比 | ≥ 95% |
| **纪要归档率** | 在 T+3 内归档纪要的会议占比 | ≥ 90% |
| **全流程达标率** | 4 个节点全部按时完成的会议占比 | ≥ 75% |

## 输入数据

### 数据源 1：组织人员信息与运作规则（MCP Server）

社区包含多种组织类型：**SIG、TC、TSC、理事会、专家委员会、用户委员会、技术治理委员会**等，且未来可能新增其他组织类型。为统一获取完整的组织结构和人员数据，采用 **Model Context Protocol（MCP）** 方案，由社区基础设施侧提供 MCP Server。

社区仓库中（例如：Ascend/community）由运营人员统一补充社区全量组织、人员、会议运作信息，供MCP Server提供数据来源。

#### 方案说明

社区基础设施团队实现一个 **Community Organization MCP Server**，遵循 [Model Context Protocol](https://modelcontextprotocol.io/) 规范，将社区组织数据封装为标准化的 Tool 供本系统（及 AI Agent）调用。

**MCP Server 需暴露的核心 Tools：**

| Tool 名称 | 描述 | 参数 | 返回 |
|-----------|------|------|------|
| `list_org_types` | 列出社区所有组织类型 | 无 | 组织类型列表，如 `["board", "expert_committee", "user_committee", "tsc", "tc", "sig", ...]` |
| `list_organizations` | 列出指定类型下的所有组织 | `org_type: string` | 组织列表，含名称、ID、所属项目等 |
| `get_org_detail` | 获取单个组织的详细信息 | `org_id: string` | 成员列表（含角色）、会议频率要求、meeting_url、Etherpad 链接等 |
| `get_org_members` | 获取组织成员及角色 | `org_id: string` | 成员列表，含 `role`（chair / maintainer / member 等） |
| `get_meeting_rules` | 获取组织的会议运作规则 | `org_id: string` | 会议频率、议程要求、纪要要求等 |

**MCP 可行性评估：**

- ✅ **可行**：MCP 协议本身是轻量级 JSON-RPC over HTTP/SSE，社区基础设施只需实现一个服务端，将已有的 `org-info.yaml`、`sig-info.yaml` 以及其他组织数据封装为 Tool 即可
- ✅ **解耦**：本系统不依赖特定数据存储格式（YAML/数据库），只面向 MCP Tool 接口编程
- ✅ **可扩展**：新增组织类型（如未来增加"工作组"等）只需 MCP Server 侧更新数据，本系统零改动
- ✅ **AI 友好**：天然支持 AI Agent 通过 MCP 查询组织数据并做智能分析
- ⚠️ **前置依赖**：需社区基础设施团队配合开发 MCP Server，建议先实现只读 Tools，后续可扩展写入能力

### 数据源 2：会议记录（MCP Server）

由社区基础设施团队实现 **Ascend Meeting MCP Server**，将 Ascend Meeting Center 的查询接口封装为标准 MCP Tools，供本系统及 AI Agent 调用。

**MCP Server 需暴露的核心 Tools：**

| Tool 名称 | 描述 | 参数 | 返回 |
|-----------|------|------|------|
| `list_meeting_schedules` | 查询指定组织在指定周期内的会议预约记录 | `org_id: string, period: string` | 会议列表，含时间、主题、参会人 |
| `get_meeting_attendance` | 查询指定会议的实际出席情况 | `meeting_id: string` | 实际参会人数、出席名单 |
| `get_last_meeting` | 查询组织最近一次已召开的会议 | `org_id: string` | 会议时间、主题、出席人数 |

**封装价值：**

- ✅ **接口隔离**：Meeting Center REST API 变更时只需 MCP Server 侧适配，本系统调用方式不变
- ✅ **语义化查询**：AI Agent 可通过自然语言意图映射到具体 Tool，无需理解底层 REST URL 规范
- ✅ **统一接入层**：与数据源 1 架构一致，所有组织数据通过 MCP 统一获取

### 数据源 3：会议纪要（MCP Server）

由社区基础设施团队实现 **Etherpad MCP Server**，将 Etherpad 的内容查询接口封装为 MCP Tools 和 **MCP Resources**，供本系统及 AI Agent 调用。

**MCP Server 需暴露的核心 Tools 和 Resources：**

| 类型 | 名称 | 描述 | 参数 | 返回 |
|------|------|------|------|------|
| Tool | `get_pad_content` | 获取指定 pad 的完整内容 | `pad_name: string` | pad 全文（Markdown / 纯文本） |
| Tool | `get_pad_last_updated` | 获取 pad 最后更新时间 | `pad_name: string` | 最后修改时间戳 |
| Tool | `list_org_pads` | 列出组织关联的所有 pad | `org_id: string` | pad 名称列表及对应会议日期 |
| Resource | `pad://{pad_name}` | 将 pad 内容作为 MCP Resource 暴露 | — | AI Agent 可直接订阅内容变更 |

**纪要数据适合 MCP 的特殊价值：**

- ✅ **LLM 直接可消费**：`get_pad_content` 返回的非结构化文本可直接传入 LLM，用于自动判断纪要是否包含出席人、讨论摘要、决议事项等质量要素，是三个数据源中 AI 替代人工最直接的场景
- ✅ **Resource 类型支持订阅**：pad 作为 MCP Resource 暴露后，AI Agent 可主动感知纪要更新，实现"会后自动触发纪要质量检查"而无需轮询
- ✅ **架构一致**：与数据源 1、2 统一通过 MCP 接入，系统无需维护三套不同的 HTTP 客户端

### 数据源 4：邮件列表（MCP Server）

由社区基础设施团队实现 **Mailing List MCP Server**，用于检测各组织是否按时发送了议题征集邮件（对应会议生命周期节点 ②）。

**MCP Server 需暴露的核心 Tools：**

| Tool 名称 | 描述 | 参数 | 返回 |
|-----------|------|------|------|
| `search_emails` | 搜索指定邮件列表中符合条件的邮件 | `list_id: string, keywords: string[], date_from: string, date_to: string` | 邮件列表，含发件人、主题、时间 |
| `get_email_detail` | 获取单封邮件详情 | `email_id: string` | 邮件全文 |
| `list_mailing_lists` | 列出组织关联的邮件列表 | `org_id: string` | 邮件列表地址 |

**检测逻辑：**

- 在会议日 T-5 天时，系统调用 `search_emails` 查询该组织邮件列表中是否存在包含"议题征集"、"agenda"、"topic collection"等关键词的邮件
- 若未检测到匹配邮件，判定节点 ② 未达成（`TOPICS_MISSING`），触发议题征集超期提醒

**封装价值：**

- ✅ **节点 ② 检测自动化**：无需人工确认议题征集邮件是否已发送，系统自动判定
- ✅ **AI 可分析邮件内容**：LLM 可通过 `get_email_detail` 分析征集邮件质量（是否包含具体议题、截止时间等）
- ✅ **架构一致**：与数据源 1、2、3 统一通过 MCP 接入

### 输出 2：自动提醒消息

> 说明：提醒规则围绕会议生命周期的 4 个关键节点设计。系统每日检查各组织当前会议实例的节点状态，未按时完成的节点立即触发对应提醒。每个组织均配置有对接的**社区运营人员**（`operators` 字段），从中期预警起社区运营介入推进。

#### 节点级提醒

| 节点 | 事件 | 触发条件 | 提醒对象 | 渠道 | 内容模板 |
|------|------|----------|----------|------|----------|
| **① 预定会议** | 预约提醒 | 周期开始后 3 天仍未预约 | 组织负责人 | 邮件 + 消息 | "请尽快预约本周期 {组织名} 例会（建议在会议日 T-7 前完成）" |
| **① 预定会议** | 预约超期 | 距会议日 T-7 仍未预约 | 组织负责人 + 社区运营 | 邮件 + 消息 | "⚠️ {组织名} 会议预约已超期，请立即安排" |
| **② 征集议题** | 议题征集提醒 | 距会议日 T-7，已预约但未发送征集邮件 | 组织负责人 | 邮件 + 消息 | "请向邮件列表发送 {日期} 会议议题征集" |
| **② 征集议题** | 议题征集超期 | 距会议日 T-5 仍未检测到征集邮件 | 组织负责人 + 社区运营 | 邮件 + 消息 | "⚠️ {组织名} {日期} 会议的议题征集邮件尚未发出" |
| **③ 召开会议** | 会议未召开 | 会议日 T 当天结束后无出席记录 | 组织负责人 + 社区运营 | 邮件 + 消息 | "🔴 {组织名} {日期} 会议未按计划召开" |
| **④ 会议纪要** | 纪要催办 | 会议结束 T+1 天未归档 | 组织负责人 | 邮件 | "请归档 {日期} 会议纪要（截止：T+3）" |
| **④ 会议纪要** | 纪要超期 | 会议结束 T+3 天仍未归档 | 组织负责人 + 社区运营 | 邮件 + 消息 | "⚠️ {组织名} {日期} 会议纪要已超过归档期限" |

#### 周期级提醒

| 事件 | 触发条件 | 提醒对象 | 渠道 | 内容模板 |
|------|----------|----------|------|----------|
| **周期初提醒** | 会议周期起始日 | 组织负责人 | 邮件 + 群消息 | "请安排本周期 {组织名} 例会，截止日期 {周期末}" |
| **紧急预警** | 距周期末 ≤7 天仍未预约 | 组织负责人 + 社区运营 | 邮件 + 群消息 | "🔴 {组织名} 会议风险：距截止日仅剩 {N} 天" |
| **TC/委员会周期检查** | 每季度末 | TC Chair / 委员会主任 + 社区运营 | 邮件 | "本季度已召开 {N} 次会议，年度目标 ≥{M} 次" |
| **不活跃预警** | 连续 2 个周期未参加组织会议 | 成员本人 + 社区运营 | 邮件 | "根据社区规则，连续 3 个周期不参会将触发不活跃判定" |

发送提醒后，需在下次报告中记录已发送提醒次数、提醒收件人，以及各节点的当前达成状态。社区运营收到预警后应主动联系组织负责人，了解原因并协助推进。

## 处理流程

### 阶段一：数据基础建设

1. **接入 Community Organization MCP Server**：与社区基础设施团队协作，推动实现 MCP Server，提供组织结构、人员、会议规则的标准化查询能力；同步实现降级方案（直接读取 YAML 文件）
2. **标准化 meeting_url 字段**：推动各组织在对应配置文件中填写真实的 Etherpad 纪要链接，制定 PR 模板和检查规则
3. **建立会议注册表**：创建一个中心化的 `meeting-registry.yaml` 或数据库，记录每个组织的以下信息：
   - 组织 ID、类型（sig / tc / tsc / board / expert_committee / user_committee 等）、所属项目
   - 预期会议频率（monthly / quarterly / semi-annual）、会议日（如"每月第二个周三"）
   - Etherpad pad 名称、通知群组、关联邮件列表
   - **社区运营人员名单**（`operators`）：配置负责该组织会议推进的社区运营人员，含姓名、邮箱等联系方式

   示例结构：
   ```yaml
   organizations:
     - org_id: "sig-mindspeed-llm"
       org_type: "sig"
   ```

### 阶段二：监控与预警系统

1. **开发会议生命周期轮询服务**：每日检查各组织当前周期的会议实例处于哪个节点（① 预定 → ② 征集议题 → ③ 召开 → ④ 纪要），通过 4 个 MCP Server 分别检测各节点达成状态，对超期节点立即标记并触发提醒
2. **实现节点级 + 周期级提醒引擎**：按上述两级提醒规则表，在节点超期或周期超期时通过邮件/群消息通知相关人；提醒对象从 `meeting-registry.yaml` 的 `operators` 字段读取社区运营人员，确保风险预警准确送达对接运营
3. **建设仪表盘**：实现全局概览、生命周期看板、组织明细、风险预警、委员会追踪等核心视图，支持按节点状态筛选和排序

### 阶段三：治理流程闭环

1. **定义升级流程**：
   - 第 1 次未达标 → 邮件提醒 + 记录
   - 连续 2 次未达标 → 社区运营介入，联系组织负责人沟通原因
   - 连续 3 次未达标 → 社区运营上报给对应上级组织审议（如 SIG 上报 TC，TC 上报 TSC），考虑角色调整（对应社区规则"3 个月不活跃"）
2. **制定议程/纪要质量标准**：定义最低内容要求（如：议程需包含至少 1 个讨论议题；纪要需包含出席人、讨论摘要、决议事项）
3. **自动月报生成**：每月 1 日自动生成上月报告，发送给 TSC、各 TC Chair、各委员会主任及对接社区运营

### 阶段四：持续优化

1. **收集反馈**：每季度征集 Maintainer 对提醒机制的反馈，调整频率和措辞
2. **扩展指标**：增加会议参与人数趋势、议题完成率等进阶指标
